# Configuration

This page records the v1.0 configuration surfaces that are wired today and the
release modes that must be named in artifacts.

## Session settings

Session-local runtime parameters use PostgreSQL-style `SET`, `SHOW`, and
`RESET`, plus the compatibility spelling documented in
[`SET VARIABLE`](sql/set-variable.md). Supported built-in settings keep their
existing validation, and dotted custom settings such as `ultrasql.tenant` are
stored per connection.

## Memory

- `work_mem` appears through `pg_catalog.pg_settings` for client tooling.
- Per-query memory enforcement and spill behavior are tracked separately under
  the release hardening backlog.
- `ULTRASQL_TPCH_POOL_FRAMES` tunes the TPC-H benchmark buffer-pool frame count
  used by benchmark harnesses.
- `ULTRASQL_PAGE_SPILL_BACKING` selects the page-spill backing path used by
  storage tests and benchmark profiles.

## WAL

- `ULTRASQL_DATA_DIR` or `ultrasqld --data-dir DIR` enables WAL-backed storage.
- `ultrasqld --wal-sync-method fsync|fsync_writethrough` (env
  `ULTRASQL_WAL_SYNC_METHOD`) selects the durability flush primitive,
  mirroring PostgreSQL's `wal_sync_method`. The default `fsync` issues
  `fsync(2)` on every durability barrier — the same class PostgreSQL
  (`fsync = on` with its default `wal_sync_method`) and SQLite
  (`synchronous=FULL` with default `fullfsync` off) provide. On Linux this
  forces the device write cache, so completed commits survive power loss. On
  macOS, `fsync(2)` does not force the drive's own cache — identical to the
  PostgreSQL/SQLite default posture there — so commits survive an OS crash but
  sudden power loss can lose or reorder whatever is still in the drive's
  volatile cache (not only the newest commits: unordered destage can persist a
  later barrier while an earlier cached write is lost, the same exposure
  PostgreSQL documents for its macOS default); set `fsync_writethrough`
  (PostgreSQL's name for `fcntl(F_FULLFSYNC)`) to force the drive cache to
  stable media at a substantial per-commit latency cost (~60x on Apple SSDs).
- `ultrasql --archive-wal WAL_PATH --archive-dir DIR` copies one WAL file into
  an archive directory.
- `ultrasql --restore-wal WAL_NAME --archive-dir DIR --restore-output PATH`
  restores one archived WAL file.
- `ultrasql --wal-send-once DEST --archive-dir DIR --data-dir DATA_DIR` and
  `ultrasql --wal-receive-once SOURCE --data-dir DATA_DIR` exercise file-backed
  physical shipping. Continuous sender/receiver loops remain release hardening
  work.

## Object-store

Object scans use explicit SQL paths and object-store table functions. The
object Parquet range benchmark records remote bytes, range request count,
cache hits, and cache misses through:

```bash
benchmarks/object_parquet_range.sh
```

Set `OBJECT_PARQUET_RANGE_PROFILE=smoke|full` and
`OBJECT_PARQUET_RANGE_OUT_DIR=benchmarks/results/latest` for the committed
runner. Object-store paths must stay explicit; do not hide a whole-object read
behind a range artifact.

## COPY

Binary `COPY` — both `COPY … FROM <file> WITH (FORMAT binary)` and
`COPY … FROM STDIN WITH (FORMAT binary)` — is bounded to **128 MiB per stream**
by default to prevent an unbounded-buffer OOM. Override the ceiling with the
`ULTRASQL_COPY_BINARY_FILE_LIMIT_BYTES` environment variable (a positive byte
count; a zero or invalid value falls back to the 128 MiB default). The STDIN
path rejects the transfer once the cumulative `CopyData` size would exceed the
limit; the file path rejects oversized files before reading.

## ANN

HNSW and IVFFlat indexes are created with SQL syntax:

```sql
CREATE INDEX docs_embedding_hnsw ON docs USING hnsw (embedding vector_l2_ops)
    WITH (payload = bf16);

CREATE INDEX docs_embedding_ivfflat ON docs USING ivfflat (embedding vector_l2_ops)
    WITH (lists = 64, probes = 8, payload = int8);
```

HNSW accepts the `payload` option in the current SQL path. IVFFlat accepts
`lists`, `probes`, and `payload`. The design notes describe `ef_search`; the
release benchmark runners record their own ANN search parameters in artifacts.

## Benchmark modes

- PR gate: `benchmarks/certify.sh smoke`
- Manual/full gate: `benchmarks/certify.sh full`
- TPC-H SF10: `benchmarks/tpch_sf10_certify.sh`
- ClickBench: `benchmarks/clickbench_certify.sh`
- Firebolt Core local: `benchmarks/firebolt_core_local.sh start`
- Firebolt Core endpoint override, internal only:
  `FIREBOLT_CORE_ENDPOINT=http://127.0.0.1:3473`
- Hot-path flamegraphs: `benchmarks/hot_path_profiles.sh`
- Backup/restore smoke: `benchmarks/backup_restore_smoke.sh`

Benchmark artifacts must record host metadata and must not claim a win when a
measured engine is `not_available`.

## RLS

RAG tenant security uses visible row-level security policy SQL and the
`ultrasql.tenant_id` setting generated by catalog helper functions. The current
scope is a database-enforced tenant slice with persisted policies plus owner,
superuser, and `BYPASSRLS` bypass behavior. Broader RLS policy combinations
still track role-qualified policies and wider write-path checks. See
`docs/rag-tenant-security.md`.

## Ops

- `ULTRASQL_OPS_LISTEN` or `ultrasqld --ops-listen ADDR` exposes `/health`,
  `/ready`, and `/metrics`.
- `ULTRASQL_OPS_TOKEN` or `ultrasqld --ops-token TOKEN` enables
  `/backup/start` and `/backup/stop`; callers must send
  `Authorization: Bearer TOKEN`.
- `ultrasql --ops-endpoint ADDR --isready` checks readiness through the ops
  endpoint.
- `ultrasql --data-dir DIR validate` checks catalog, indexes, WAL, heap
  visibility, and ANN tombstones.
